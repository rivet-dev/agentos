When a RivetKit actor sleeps, its VM is destroyed; on wake a fresh VM boots and the actor's durable
SQLite is replayed. The live agent session does **not** survive that teardown — it is recreated on
demand. Session resume restores continuity so you can keep prompting the same `sessionId` as if
nothing happened.

## What survives a sleep/wake

| Thing | Survives? | Where |
| --- | --- | --- |
| Guest filesystem | ✅ | actor SQLite (`agent_os_fs_entries`, via sqlite_vfs root) |
| Session registry (which sessions exist + capabilities) | ✅ | actor SQLite (`agent_os_sessions`) |
| Conversation transcript | ✅ | actor SQLite (`agent_os_session_events`) |
| The live agent process / in-memory context | ❌ | recreated on wake |

The stable, client-facing `sessionId` is preserved across wakes. Internally the actor may map it to a
freshly-created live session, but you never see that.

## How resume works

Resume is **lazy**: it runs the first time you prompt a session that exists in the registry but is
not currently live. It is also **capability-driven** — the orchestrator never special-cases an agent
by name, only by what the agent advertises.

1. **Native resume (optimization).** If the agent advertises ACP `loadSession`/`resume` (e.g.
   OpenCode, whose store persists under `$HOME` on the durable root), the actor issues
   `session/load`. The agent restores its own context. The `sessionId` is unchanged.
2. **Universal fallback.** If the agent has no native resume (e.g. Pi), or its store didn't survive,
   the actor reconstructs a Markdown transcript from the recorded events, writes it into the VM (for
   example `/root/.agentos/threads/<sessionId>.md`), starts a fresh session, and prefixes your next
   prompt with a pointer such as *"Continuing thread at `<path>` — read it if you need prior
   context."* The agent reads the transcript with its normal file tools.

Because the fallback only needs file-read tools, it works for **any** agent — no per-agent code.

<Aside type="note">
The fallback is intentionally "lossy but universal": the agent is *pointed at* the transcript rather
than having it pre-loaded into its context window. Agents that support native ACP resume get
full-fidelity restoration instead.
</Aside>

## Transcript capture

Every turn is recorded to the actor's SQLite as it happens — the outbound prompt text and the
inbound ACP `session/update` stream (agent messages and tool calls). The transcript file handed to
the agent on a fallback is a disposable render of that canonical log, rebuilt fresh each resume.

<Aside type="note">
The recorded `seq` is internal transcript ordering only. The live `sessionEvent` subscription stays
live-only — there is no client-facing replay cursor or event recovery channel.
</Aside>

## Durability requirement (direct SDK)

Resume depends on a **durable root filesystem**. The RivetKit actor configures one automatically
(its sqlite_vfs root). A direct `AgentOs` SDK user on the default in-memory root has no durable
store, so transcript capture is a no-op and resume cannot restore context there. If you need resume
outside the actor, configure a durable root explicitly.
